## 1장 유닛 테스트의 목표

### 유닛 테스트의 본질적인 목표

- 지속 가능한 소프트웨어 성장이 유닛 테스트의 궁극적인 목적.
- 더 나은 설계는 부수적인 효과일 뿐이며, 핵심 목표는 아님.
- 테스트는 코드의 **엔트로피 증가**(복잡도, 혼란)를 막기 위한 안전망 역할을 수행.

### 나쁜 테스트와 좋은 테스트

- 테스트는 단순히 많이 작성한다고 효과적이지 않음.
- 유지 비용이 높은 테스트는 순 가치가 0 혹은 음수일 수 있음.

**좋은 테스트는 다음을 만족해야 함**

- 유지보수 비용이 낮음
- 회귀(regression)가 발생했을 때 문제를 조기에 탐지할 수 있도록 도움
- 핵심 도메인 로직을 정확히 타겟팅한다

**NOTE**

> **Regression**
 : 기존에 잘 동작하던 기능이, 새로운 코드 변경이나 기능 추가 후에 의도치 않게 망가지는 현상

### 테스트 커버리지의 맹점

**Code Coverage**

- 단순히 코드 라인이 실행되었는지를 기준으로 측정.
- 삼항 연산자 사용 등으로 수치를 조작할 수 있음.
- 높은 커버리지 수치가 좋은 테스트를 의미하진 않음.

**Branch Coverage**

- 분기 경로를 모두 테스트했는지를 측정.
- 더 정밀하지만, 여전히 assertions 부재나 외부 라이브러리 내부 경로는 고려하지 못함.

**요약**

- Coverage는 negative indicator다.
- 커버리지가 낮으면 문제가 있는 게 맞지만, 커버리지가 높다고 해서 좋은 테스트라는 보장은 없다.

### 성공적인 테스트 스위트의 조건

- 개발 주기와 통합되어야 함
→ 모든 변경에 대해 테스트가 실행돼야 의미 있음.

- 중요한 코드 영역만 타겟팅
→ 주로 도메인 로직 (비즈니스 규칙)에 집중해야 함.

- 최소 비용으로 최대 가치 제공
→ 테스트 자체의 설계와 유지보수 효율성 중요.
